--- AUTH API ROUTES ---

POST /api/auth/organizer/register
  - Validates fields, checks email uniqueness, hashes password, inserts user, returns JWT cookie

POST /api/auth/organizer/login
  - Verifies password hash, sets ventry_token cookie, returns user data

POST /api/auth/admin/login
  - Checks against ADMIN_EMAIL / ADMIN_PASSWORD env vars (no DB row for admin)

POST /api/auth/logout
  - Clears ventry_token cookie (maxAge: 0)

GET  /api/auth/me
  - Returns organizer or admin profile based on role in token

POST /api/auth/forgot-password
  - Creates reset_tokens DB entry, sends password reset email via Resend
  - Always returns success (avoids email enumeration)

POST /api/auth/reset-password
  - Verifies JWT token + DB entry, updates password hash, marks token used

POST /api/auth/change-password
  - Authenticated route, verifies current password, updates hash

--- PUBLIC API ROUTES ---

GET  /api/events
  - Query params: q (search), category, city
  - Returns approved events with organizer join and tiers

GET  /api/events/[id]
  - Returns single event with full details

POST /api/checkout
  - Verifies event approved + tier availability
  - Calls Paystack initializeTransaction
  - Creates pending_orders record (non-critical)
  - Returns authorizationUrl for Paystack redirect

POST /api/paystack/webhook
  - Verifies HMAC-SHA512 signature
  - Processes charge.success events only
  - Idempotency check on paystack_reference
  - Generates ticketId (TKT-XXXX-XXXX), refundCode (RF-XXXX-XX), qrToken (JWT)
  - Generates QR data URL, inserts ticket, increments tier sold count
  - Creates/updates payout record for the organizer
  - Sends ticket confirmation email with QR code embedded

GET  /api/paystack/callback
  - Redirects to /payment/pending?ref=... (webhook is source of truth)

GET  /api/tickets/[id]
  - Fetches ticket with event + organizer + tier joins
  - Generates QR data URL on demand from stored qr_token

POST /api/tickets/retrieve
  - Looks up ticket by id + email pair (buyer verification)

GET  /api/tickets/by-reference
  - Polls for ticket by paystack_reference (used by /payment/pending page)

POST /api/help/refund
  - Verifies ticket + refundCode + email
  - Checks for existing open complaint
  - Inserts complaint record with priority (Fraud = high)

--- ORGANIZER API ROUTES ---

GET  /api/organizer/me
  - Returns organizer profile from users table

GET  /api/organizer/stats
  - Returns ticketsSold, revenueInEscrow, activeEvents, payoutDue

GET  /api/organizer/events
POST /api/organizer/events
  - GET: filtered by status, returns tiers
  - POST: multipart form with banner upload to Supabase Storage (event-assets bucket)
  - Requires kyc_status = approved before allowing event creation

PATCH /api/organizer/events/[id]
DELETE /api/organizer/events/[id]
  - Both verify organizer_id ownership before acting

POST /api/organizer/scan
  - Five-check QR validation:
    1. JWT signature valid
    2. Ticket exists in DB
    3. Ticket belongs to correct event
    4. Status is 'valid' (not already used/refunded/cancelled)
    5. Event not expired more than 24h ago
  - Marks ticket status = 'used', logs scan to scan_logs table

GET  /api/organizer/scan/logs
  - Returns scan logs + stats (scanned, rejected) for a given eventId

GET  /api/organizer/payouts
  - Returns payout history for organizer

POST /api/organizer/bank
  - Validates 10-digit account number format
  - Saves bank_name, account_number, account_name to users table

PATCH /api/organizer/settings
  - Updates name, phone, bio, notification preferences

POST /api/organizer/kyc
  - Multipart, handles steps 1–5:
    Step 1: gov ID + selfie (upload to kyc-documents bucket, images only, max 5MB)
    Step 2: phone number
    Step 3: social media handles
    Step 4: venue proof document (images + PDF, max 5MB)
    Step 5: final submission → sets kyc_status = 'pending'
